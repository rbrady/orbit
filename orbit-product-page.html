orbit

Secret rotation without the panic. Provision, rotate, and revoke credentials across your entire infrastructure from one command.

Get Started View on GitHub →
$ cargo install orbit ⎘ copy

The problem

Incidents don't wait for runbooks.

When a secret is compromised, every minute it stays active is a minute of exposure. Manual rotation across multiple services, environments, and backends is slow, error-prone, and stressful under pressure.

⏱

Slow under pressure

Cross-referencing runbooks while the clock ticks introduces mistakes. Missed services mean incomplete rotation.

🗺

Scattered knowledge

Where does each secret live? Who created it? How do you revoke it? That knowledge shouldn't live in someone's head or a stale wiki.

🔁

No safe iteration

Without a dry-run, you can't verify what will change before changing it. One mistake during a rotation can make things worse.

📋

Audit gaps

Manual processes leave no record of what was rotated, when, and by whom — exactly the information an incident review needs.

How it works

Three steps. One command.

orbit uses declarative config files that live alongside your service definitions. Define once, rotate whenever.

1

Define

Drop an orbit.json next to your service config. Declare the secret type, its source URL, and where to store it. Auth comes from your ambient environment — no credentials in the file.

2

Verify

Run orbit rotate --dry-run to see exactly what will change before anything touches production. Inspect scope across a single file or an entire directory tree.

3

Rotate

Drop --dry-run to execute. orbit creates the new secret, stores it in your backend, then revokes the old one. Clean, atomic, logged.

One file. All the context.

An orbit.json file lives alongside the service it belongs to in your repo. It describes what to rotate and where to store it — never how to authenticate.

  • Auth is ambient. orbit resolves credentials from your environment — setup-context, VAULT_TOKEN, gcloud, AWS — and only prompts when it can't find what it needs.
  • Multiple secrets per file. Group related secrets (e.g. all credentials for a Tanka environment) and rotate them as a batch, or target a single one with --secret <name>.
  • No credentials in git. The config describes the secret's location and type — values never touch the file.
  • Generate with a wizard. Run orbit generate to create a config interactively without touching JSON by hand.

Commands

Everything you need for the full lifecycle.

orbit provision [file|path] [--dry-run]

Idempotent bootstrap. Creates and stores secrets that don't exist yet. Skips anything already provisioned. Safe to run repeatedly.

orbit rotate [file|path] [--dry-run]

Creates a new secret, stores it in your backend, then revokes the old one. Always produces a net-new credential.

orbit revoke [file|path] [--dry-run]

Immediately invalidates the current secret at the source and removes it from storage. No replacement — use during containment.

orbit generate

Interactive wizard that asks questions about your secret type and storage backend, then writes a valid orbit.json for you.

orbit list [file|path]

Read-only summary of all secrets orbit knows about in the given path. No API calls. Great for auditing coverage.

orbit validate [file|path]

Schema check only. Confirms every orbit.json is well-formed and uses a registered adapter type.

Adapters

Works with the secrets you already have.

orbit ships with adapters for the most common secret types, with Vault and 1Password as storage backends.

Secret types

TypeAuth
grafana-access-policy-token env: GRAFANA_API_TOKEN
gcom-api-token env: GRAFANA_COM_TOKEN
aws-iam-access-key ambient AWS credentials
gcp-service-account-key ambient gcloud auth
slack-token prompted / 1Password

Storage backends

BackendAuth
vault VAULT_TOKEN or ambient vault login
1password op CLI / OP_SERVICE_ACCOUNT_TOKEN

Extensibility

Adding a new secret type takes one file.

orbit's adapter model is built on Rust traits. Implement RotateSecret, register it, and your new type is available to every orbit command. The traits are intentionally small.

Adapters that only support creation (write-once tokens with no revocation API) implement the lighter ProvisionSecret trait and work with orbit provision.

◈ src/adapters/my_token.rs 
impl RotateSecret for MyTokenAdapter {
  fn type_name() -> &'static str {
    "my-token-type"
  }

  fn create(
    &self,
    config: &AdapterConfig,
    auth: &AuthContext,
  ) -> Result<Secret> {
    // call your API here
    let token = api::create_token(&config.url, &auth.token);
    Ok(Secret::new(token))
  }

  fn revoke(&self, secret: &Secret, ...) -> Result<()> {
    api::delete_token(&secret.value)
  }
}
